## 1. プロジェクト概要

### 1.1 コンセプト

GeoGraphy は「No-Texture・Plugin駆動・マルチライブラリ対応」の映像制作プラットフォームです。

単なる VJ ツールではなく、複数の描画ライブラリを Plugin として束ねるプラットフォームとして設計されています。Three.js・PixiJS・opentype.js など、異なるライブラリを Plugin Group として追加でき、ユーザーは必要な Plugin だけを有効化して自分専用の最小構成で起動できます。

アプリ名は「Geometry（幾何学）+ 地形（Geo）」と「Graph（グラフ・図形）+ 描写（Graphy）」のダブルミーニングです。

### 1.2 ビジュアルコンセプト

- レトロフューチャー・シンセウェーブ
- ワイヤーフレーム・グリッド・幾何学形状
- テクスチャを一切使わない・計算とベクターだけで完結
- 暗背景 + 発光するライン・形状・面

### 1.3 Goals

- Three.js によるリアルタイム 3D レンダリング
- Electron アプリとしてネイティブ配布（.dmg）
- Plugin Group 単位でライブラリを追加できるプラットフォーム設計
- CLI ツールで新規 Plugin を自動スキャフォールド
- MIDI コントローラーによるリアルタイムパラメーター操作
- Resolume Arena との連携を前提とした設計
- Program / Preview バス構造による本格的な映像ミキシング
- MixerPlugin による操作 UI の完全カスタマイズ（v2〜）
- Transition Plugin による表現変形エンジン
- Geometry Launcher・Sequencer による自動 VJ 運用（v2〜）
- Embed Export：映像を Web サイト・スクリーンセーバーに埋め込む（v2〜）
- OSS として公開・世界中のコントリビューターが Plugin を追加できる

### 1.4 非機能要件・対象外

**全バージョン対象外（変更不可原則）：**
- テクスチャ貼り付け（No-Texture 原則・全バージョン禁止）
- 動画再生（Resolume に任せる）
- クラウド同期・コラボレーション
- Tauri / ネイティブアプリ化（Electron で解決済み）
- スマートフォン最適化（v2 以降）

**将来検討（v4 以降）：**
- Claude.ai 自動トランジション
- Video Input Driver（Resolume 受信）

## 3. Electron アーキテクチャ（Day24〜27 確立）

### 3.1 採用理由

| 比較 | Web ホスティング | Electron |
|---|---|---|
| サーバー | 必要 | **不要** |
| 配布方法 | URL | **.dmg / .exe をダウンロード** |
| ファイル操作 | ブラウザ制限あり | **Node.js fs で自由** |
| プラグイン追加 | 困難 | **外部フォルダをスキャン可能** |
| メニューバー | なし | **ネイティブメニューバー** |

### 3.2 プロセス構成

```
electron/
  main.js       ← メインプロセス（ウィンドウ生成・ファイルアクセス・メニュー）
  preload.js    ← レンダラーと main の橋渡し（IPC・contextBridge）
src/            ← 既存 Vite + React コードそのまま
```

セキュリティ設定（必須）：
- `nodeIntegration: false`
- `contextIsolation: true`
- `sandbox: true`

### 3.3 Electron 固定部分（レンダラーが触れない）

| 処理 | 場所 | 理由 |
|---|---|---|
| メニューバーの定義（GeoGraphy / File / View） | main.js | Electron `Menu` API |
| ファイルの Save / Load / Save As... | main.js | Node.js fs 操作 |
| ネイティブダイアログ（open/save） | main.js | Electron `dialog` API |
| Recent ファイルの管理・永続化 | main.js | ファイルシステム管理 |
| 外部プロセス連携・ファイル書き出し | main.js | Node.js 領域 |

### 3.4 レンダラー（React）に残すべき処理

| 処理 | 理由 |
|---|---|
| engine への操作（restoreProject / buildProject） | Three.js と密結合 |
| UI 状態管理 | React の責務 |
| Web MIDI API（MidiInputWrapper 経由） | ブラウザ API |
| autosave の送受信ロジック（useAutosave.ts） | engine と密結合 |

### 3.5 メニューバー設計

| メニュー | 項目 |
|---|---|
| GeoGraphy | About / Preferences...（⌘,）/ Services / Hide / Quit |
| File | New（⌘N）/ Open...（⌘O）/ Open Recent > / Save（⌘S）/ Save As...（⌘⇧S）/ Close Window |
| **View** | **全 Window の表示/非表示トグル（後述 §5）** |

### 3.6 将来実装時の注意（Output タブ）

Output タブで外部出力（録画・NDI・Syphon など）を実装する際は、最初から main.js 経由（IPC）で設計すること。レンダラーに Node.js 操作を直書きしない。

### 3.7 isDev 判定（MUST）

```javascript
// ✅ 正しい判定
const isDev = !app.isPackaged

// ❌ 使用禁止（Electron ビルドで信頼できない）
// const isDev = process.env.NODE_ENV !== 'production'
```

### 3.8 electron/ ディレクトリは CommonJS

`electron/package.json` に `{ "type": "commonjs" }` を置く。ルートの `"type": "module"` と共存するための必須対応。

## 5. Window / Panel 設計（v1.9 新設）

### 5.1 命名原則

| 名称 | 定義 | 例 |
|---|---|---|
| **Window** | Plugin エコシステムの UI・コントリビューターがデザインできる | Mixer Simple Window / FX Simple Window / Window Plugin |
| **Panel** | アプリ固定の小窓・設定や選択のための UI・コントリビューターが触れない | Preferences Panel |

### 5.2 Simple Window（v1.9 新概念）

各 Plugin にデフォルトで付属する最小 UI。フォールバックとして機能する。

- **カスタム Window Plugin が有効なとき** → Simple Window は非表示
- **カスタム Window Plugin がないとき** → Simple Window が表示される

| Simple Window 名 | 対応 Plugin | 旧名称 |
|---|---|---|
| Mixer Simple Window | MixerPlugin | SimpleMixer |
| FX Simple Window | FXPlugin | FxControlPanel |
| Macro Knob Simple Window | MacroKnobManager | MacroKnobPanel |

### 5.3 Window Plugin（v1.9 再定義）

各 Plugin に対してカスタム UI を差し込める仕組み。

- 事前にどの Plugin にデータを渡すかを宣言して設計する
- コントリビューターが開発・PR で追加できる
- 有効化されると対応する Simple Window は非表示になる

### 5.4 View メニュー

すべての Window は View メニューから表示/非表示を切り替えられる。キーボードショートカット（§13）とも連動する。

```
View
  ├── Mixer Simple Window     （⌘1）
  ├── FX Simple Window        （⌘2）
  ├── Macro Knob Simple Window（⌘3）
  ├── Camera Simple Window    （⌘4）
  ├── ─────────────────────
  ├── Hide All Windows        （H）
  └── Show All Windows        （S）
```

## 8. プラットフォームアーキテクチャ

### 8.1 基本思想

GeoGraphy は「複数の描画ライブラリを Plugin として束ねるプラットフォーム」。エンジンは各 Plugin Group のライブラリを知らず、共通 Interface を通じてのみ通信する。

### 8.2 Plugin Group の概念

| Plugin Group | ライブラリ | バージョン | 主な用途 |
|---|---|---|---|
| Three.js Group | Three.js | v1 | 3D幾何学・ワイヤーフレーム・VJ映像 |
| PixiJS Group | PixiJS | v2 | 2D高速描画・フラットシェーディング |
| Typography Group | opentype.js | v3 | フォントアウトライン・Extrude・文字アニメーション |

### 8.3 PluginBase Interface（全 Plugin 共通）

```typescript
interface PluginBase {
  id: string
  name: string
  renderer: 'threejs' | 'pixijs' | 'opentype' | string
  enabled: boolean
}
```

### 8.4 Plugin 自動登録

```typescript
const modules = import.meta.glob('./*/index.ts', { eager: true });
Object.values(modules).forEach((m: any) => registry.register(m.default));
```

## 10. Transport Architecture（Day58 確立）

### 10.1 設計原則

GeoGraphy の全パラメーター受け渡しは **TransportEvent** フォーマットに統一されている。プロトコル（MIDI / OSC 等）の差異は Input Wrapper が吸収し、Engine はプロトコルを一切知らない。

```
【外側の世界】プロトコル・セマンティクス
  cc-mapping.md（人間・AI が読む）
  MidiInputWrapper（MIDI → TransportEvent に変換）
  将来: OscInputWrapper / AI Layer

【境界線】
  TransportEvent { slot, value, source?, time? }
    slot:  CC Standard 番号（プロトコル非依存の抽象 ID）
    value: 0.0〜1.0（正規化済み）

【コア層】純粋な値の処理（自律的に動く）
  TransportManager（slot + value を受け取るだけ）
  ParameterStore（slot番号をキーとして保持）
  Engine（CC番号の意味を知らない）
  TransportRegistry（slot→paramId の対応表）

【UI 層】鏡
  App.tsx（TransportRegistry を購読するだけ）
  WindowPlugin（availableParameters を filter するだけ）
```

### 10.2 全入力経路の統一

```typescript
// 全ての入力源が同じ入り口を通る
engine.handleMidiCC(TransportEvent)

// 物理MIDI コントローラー
MidiInputWrapper → engine.handleMidiCC({ slot, value, source: 'midi' })

// SimpleWindow スライダー
onChange → engine.handleMidiCC({ slot: ccNumber, value: normalized, source: 'window' })

// 将来: Sequencer / LFO / AI
engine.receiveMidiModulation(knobId, value)
```

### 10.3 CC Map 翻訳機

```
docs/spec/cc-mapping.md（人間が書く・セマンティック情報あり）
  ↓ pnpm gen:cc-map
settings/cc-map.json（機械が読む・意味を知らない）
  ↓ engine.initialize() 内で読み込み
TransportRegistry（コア層・slot→paramId の対応表）
```

この「外側と内側の分離」が将来の自然言語 AI 対応の基盤になる。

## 12. Program / Preview バス構造（Day25 実装済み）

### 12.1 概要

| バス | 役割 | 実体 |
|---|---|---|
| Program バス | 今出力中の映像・Resolume 等に送られる | フルサイズ Three.js Scene |
| Preview バス | 次に出す映像・エディット中 | SceneState（JSON）+ 小サムネイルキャンバス（320×180） |

### 12.2 Preview バスのメモリー設計（MUST）

- MUST: Preview バスは `THREE.Scene` / `THREE.WebGLRenderer` を持たない
- Three.js オブジェクトは Program に昇格した瞬間に生成・切り替え後に旧 Program を `dispose()`
- SceneState の変更は Command パターン経由

### 12.3 切り替えフロー

```
Preview バスで次の映像を準備
  ↓
Mixer Window でトランジションを選択
  ↓
クロスフェーダーで切り替え実行
  ↓
Transition Plugin が SceneState を変形しながら移行
  ↓
Preview が Program に昇格・旧 Program が Preview に降格
```

| Aerial Camera | `aerial-camera` | height（CC501） |

### 20.3 Camera 自動連動ルール（Day44 確定）

Geometry を変えたとき Camera を自動連動させるが、**ユーザーが手動で Camera を変えていた場合は追従しない**。

```typescript
interface Layer {
  cameraPlugin: CameraPlugin
  isCameraUserOverridden: boolean  // true = ユーザーが手動で選択済み
}
```

- `isCameraUserOverridden = false`（デフォルト）→ Geometry 切り替え時に `defaultCameraPluginId` に従う
- `isCameraUserOverridden = true`（ユーザーが手動変更済み）→ Geometry を変えても Camera は変えない

## 21. マクロノブ・CC Standard（Day13〜58 確立）

### 21.1 概要

32個のマクロノブで複数のパラメーターを同時に操作する。1ノブに最大3パラメーターを割り当て可能。

### 21.2 責務分離（Day58 確定）

| クラス | 責務 |
|---|---|
| MacroKnobManager | 32ノブの UI 設定管理（名前・CC番号・アサイン・値キャッシュ） |
| TransportManager | 全入力の唯一の通路。CC入力の処理・rangeMap |
| TransportRegistry | slot→paramId の対応表。Plugin Apply 時に更新 |

### 21.3 データフロー

```
【物理 MIDI コントローラー】
  MIDI Controller → MidiInputWrapper（Web MIDI API）
    → rawValue / 127 で正規化
    → TransportEvent { slot, value, source: 'midi' }
    → engine.handleMidiCC()

【SimpleWindow スライダー】
  onChange → normalized → engine.handleMidiCC({ slot: ccNumber, value, source: 'window' })

【共通ルート】
  engine.handleMidiCC(TransportEvent)
    → TransportManager
    → ParameterStore（slot番号をキー）
    → engine.flushParameterStore()
    → TransportRegistry で slot→param を解決
    → plugin.params.value に反映
    → paramChangedCallback → UI 更新
```

### 21.4 CC Standard（MIDI 2.0 AC 空間）

GeoGraphy 独自の CC 番号体系。Block 単位で概念を定義する。

| Block | 名前 | 代表例 |
|---|---|---|
| 1xx | EXISTENCE | CC101=Primary Amount / CC110=Auto Rotate |
| 2xx | FORM | CC201=Density / CC203=Symmetry |
| 3xx | MOTION | CC300=Speed / CC302=Deformation |
| 4xx | COLOR | CC400=Hue / CC401=Saturation |
| 5xx | SPACE | CC500〜502=Position / CC510〜512=LookAt |
| 6xx | EDGE | CC600=Edge Strength |
| 7xx | BLEND | CC700=Blend Amount / CC701=Feedback |
| 8xx | SHADER | CC801=Draw Progress |
| 9xx | SCENE | CC900=Energy（AI 向け・将来） |

詳細: `docs/spec/cc-standard.spec.md` / 対応表: `docs/spec/cc-mapping.md`
