## 1. 開発の基本方針

### 1.1 SDD × CDD 原則

- **SDD（Spec-Driven Development）**: 実装前に必ず `docs/spec/` の spec ファイルを読む。仕様変更はコードより先に spec を修正する
- **CDD（Compiler-Driven Development）**: 完了条件は `pnpm tsc --noEmit`（型エラーゼロ）+ `pnpm test --run`（全テストグリーン）の両方通過
- **`any` 禁止**: 型エラーは自律修正する
- **プラン先行**: 実装前にプランを提示・承認を得てから実装開始

### 1.2 開発サイクル

```
1. docs/spec/[機能].spec.md を確認（または新規作成）
2. プランを提示・承認を得てから実装開始
3. 実装 → pnpm tsc --noEmit → 型エラーがあれば自律修正
4. pnpm test --run → 全グリーンになるまでループ
5. docs/progress/[task].log.md に完了を記録
6. セッション終了時に git commit + Day タグ（git tag dayN && git push origin dayN）
7. HANDOVER.md を更新
8. Obsidian dev-log に記録
```

## 4. 確立済みアーキテクチャ原則

### 4.1 engine.ts の制約（MUST）

- `engine.ts` は `App.tsx` に依存してはいけない・単体で動作できること
- Parameter Store の変更は必ず Command 経由のみ（直接代入禁止）
- Plugin には `renderer`・`enabled` フィールドを持たせること

### 4.2 Plugin 二分類（Day37 確立）

```
ModulatablePlugin（MIDI 2.0 / MacroKnob 制御可能）
  継承: GeometryPlugin / FXPlugin / ParticlePlugin / LightPlugin / CameraPlugin / SequencerPlugin（予定）

PluginBase のみ（外部制御不要）
  継承: TransitionPlugin / WindowPlugin / MixerPlugin
```

### 4.3 Transport Architecture（Day58 確立）

```
【外側の世界】プロトコル・セマンティクス
  cc-mapping.md（人間・AI が読む）
  MidiInputWrapper（MIDI → TransportEvent に変換）
  将来: OscInputWrapper / AI Layer（自然言語 → TransportEvent）

【境界線】
  TransportEvent { slot, value, source?, time? }

【コア層】純粋な値の処理
  TransportManager（slot + value を受け取るだけ）
  ParameterStore（slot番号をキーとして保持）
  Engine（CC番号の意味を知らない）
  TransportRegistry（slot→paramId の対応表）

【UI 層】鏡
  App.tsx（TransportRegistry を購読するだけ）
  WindowPlugin（availableParameters を filter するだけ）
```

**全入力経路の統一:**
```typescript
// 全ての入力源が同じ入り口を通る
engine.handleMidiCC(TransportEvent)
```

**Step 4（未完）: engine の ccMapService 依存解消**
- `engine.ts` の `flushParameterStore()` が `ccMapService` を直接参照している
- これが解消されると「外側と内側の境界線」が完全に閉じる
- 影響範囲が大きいため単独セッションで設計してから実装

### 4.4 MIDI 設計原則（Day42〜58 確立）

- **外部コントローラー受信**: Web MIDI API → `MidiInputWrapper` → `engine.handleMidiCC(TransportEvent)`
- **内部バス**: `TransportEvent { slot, value: 0.0〜1.0 }` で統一（プロトコル非依存）
- **MidiInputWrapper**: `src/drivers/input/MidiInputWrapper.ts`・シングルトン `midiInputWrapper`

### 4.5 Camera Plugin 設計原則（Day44〜45 確立）

- Camera は **独立した Plugin**（Geometry に埋め込まない）
- レイヤーごとに独立したインスタンス（`getCameraPlugin()` がファクトリ呼び出し）
- `isCameraUserOverridden` フラグで自動連動とユーザー意図を両立

### 4.6 Preset 設計原則（Day47 確立）

- Preset = `GeoGraphyProject` をそのまま保存
- 保存先: `localStorage`（キー: `geography:presets-v1`）
- Load → 即 engine 反映（APPLY 不要）

### 4.7 requiresRebuild フラグ（Day46 確立）

- メッシュの頂点数・形状が変わる param には `requiresRebuild: true` を必ず設定
- `engine.setGeometryParam()` が検知して `layerManager.rebuildPlugin()` を呼ぶ（destroy → create）

### 4.8 Clock の2系統（MUST・混同禁止）

```typescript
engine.threeClock  // THREE.Clock → getDelta() でフレーム間経過時間
engine.clock       // Clock（独自）→ BPM・beat 値管理・readonly で外部公開
```

### 4.9 FX 実装ルール（MUST）

- `createFxPlugins()` ファクトリを使う（シングルトン共有禁止）
- `fxStack.update()` は `enabled=false` でもスキップしない（`pass.enabled` で制御）
- FX スタック順序は `FX_STACK_ORDER` 定数に従う

### 4.10 Window / Panel 命名原則（Day28 確立）

- **Window** = Plugin エコシステムの UI・コントリビューターがデザインできる
- **Panel** = アプリ固定の小窓（Preferences Panel など）・コントリビューターが触れない
- **Simple Window** = 各 Plugin に付属するデフォルト最小 UI・カスタム Window Plugin がないときのフォールバック

### 4.11 Electron 薄い鎖アーキテクチャ（Day77〜78 確立）

```
【鎖 1】React UI メニュー（開発・Vercel デモ用）
  ボタンクリック → Application 層（projectManager）に直結

【鎖 2】Electron ネイティブメニュー（製品・.dmg 用）
  Menu click → sendToRenderer() のみ
  → App.tsx が geoAPI 経由で Application 層を呼ぶ
  → main.js IPC ハンドラーが fs / dialog の実体を提供する

2つの鎖が同じ Application 層（projectManager）を参照する
```

## 1. 開発の基本方針

### 1.1 SDD × CDD 原則

- **SDD（Spec-Driven Development）**: 実装前に必ず `docs/spec/` の spec ファイルを読む。仕様変更はコードより先に spec を修正する
- **CDD（Compiler-Driven Development）**: 完了条件は `pnpm tsc --noEmit`（型エラーゼロ）+ `pnpm test --run`（全テストグリーン）の両方通過
- **`any` 禁止**: 型エラーは自律修正する
- **プラン先行**: 実装前にプランを提示・承認を得てから実装開始

### 1.2 開発サイクル

```
1. docs/spec/[機能].spec.md を確認（または新規作成）
2. プランを提示・承認を得てから実装開始
3. 実装 → pnpm tsc --noEmit → 型エラーがあれば自律修正
4. pnpm test --run → 全グリーンになるまでループ
5. docs/progress/[task].log.md に完了を記録
6. セッション終了時に git commit + Day タグ（git tag dayN && git push origin dayN）
7. HANDOVER.md を更新
8. Obsidian dev-log に記録
```

### 1.3 ツール役割分担

## 4. 確立済みアーキテクチャ原則

### 4.1 engine.ts の制約（MUST）

- `engine.ts` は `App.tsx` に依存してはいけない・単体で動作できること
- Parameter Store の変更は必ず Command 経由のみ（直接代入禁止）
- Plugin には `renderer`・`enabled` フィールドを持たせること

### 4.2 Plugin 二分類（Day37 確立）

```
ModulatablePlugin（MIDI 2.0 / MacroKnob 制御可能）
  継承: GeometryPlugin / FXPlugin / ParticlePlugin / LightPlugin / CameraPlugin / SequencerPlugin（予定）

PluginBase のみ（外部制御不要）
  継承: TransitionPlugin / WindowPlugin / MixerPlugin
```

### 4.3 Transport Architecture（Day58 確立）

```
【外側の世界】プロトコル・セマンティクス
  cc-mapping.md（人間・AI が読む）
  MidiInputWrapper（MIDI → TransportEvent に変換）
  将来: OscInputWrapper / AI Layer（自然言語 → TransportEvent）

【境界線】
  TransportEvent { slot, value, source?, time? }

【コア層】純粋な値の処理
  TransportManager（slot + value を受け取るだけ）
  ParameterStore（slot番号をキーとして保持）
  Engine（CC番号の意味を知らない）
  TransportRegistry（slot→paramId の対応表）

【UI 層】鏡
  App.tsx（TransportRegistry を購読するだけ）
  WindowPlugin（availableParameters を filter するだけ）
```

**全入力経路の統一:**
```typescript
// 全ての入力源が同じ入り口を通る
engine.handleMidiCC(TransportEvent)
```

**Step 4（未完）: engine の ccMapService 依存解消**
- `engine.ts` の `flushParameterStore()` が `ccMapService` を直接参照している
- これが解消されると「外側と内側の境界線」が完全に閉じる
- 影響範囲が大きいため単独セッションで設計してから実装

### 4.4 MIDI 設計原則（Day42〜58 確立）

- **外部コントローラー受信**: Web MIDI API → `MidiInputWrapper` → `engine.handleMidiCC(TransportEvent)`
- **内部バス**: `TransportEvent { slot, value: 0.0〜1.0 }` で統一（プロトコル非依存）
- **MidiInputWrapper**: `src/drivers/input/MidiInputWrapper.ts`・シングルトン `midiInputWrapper`

### 4.5 Camera Plugin 設計原則（Day44〜45 確立）

- Camera は **独立した Plugin**（Geometry に埋め込まない）
- レイヤーごとに独立したインスタンス（`getCameraPlugin()` がファクトリ呼び出し）
- `isCameraUserOverridden` フラグで自動連動とユーザー意図を両立

### 4.6 Preset 設計原則（Day47 確立）

- Preset = `GeoGraphyProject` をそのまま保存
- 保存先: `localStorage`（キー: `geography:presets-v1`）
- Load → 即 engine 反映（APPLY 不要）

### 4.7 requiresRebuild フラグ（Day46 確立）

- メッシュの頂点数・形状が変わる param には `requiresRebuild: true` を必ず設定
- `engine.setGeometryParam()` が検知して `layerManager.rebuildPlugin()` を呼ぶ（destroy → create）

### 4.8 Clock の2系統（MUST・混同禁止）

```typescript
engine.threeClock  // THREE.Clock → getDelta() でフレーム間経過時間
engine.clock       // Clock（独自）→ BPM・beat 値管理・readonly で外部公開
```

### 4.9 FX 実装ルール（MUST）

- `createFxPlugins()` ファクトリを使う（シングルトン共有禁止）
- `fxStack.update()` は `enabled=false` でもスキップしない（`pass.enabled` で制御）
- FX スタック順序は `FX_STACK_ORDER` 定数に従う

### 4.10 Window / Panel 命名原則（Day28 確立）

- **Window** = Plugin エコシステムの UI・コントリビューターがデザインできる
- **Panel** = アプリ固定の小窓（Preferences Panel など）・コントリビューターが触れない
- **Simple Window** = 各 Plugin に付属するデフォルト最小 UI・カスタム Window Plugin がないときのフォールバック

### 4.11 Electron 薄い鎖アーキテクチャ（Day77〜78 確立）

```
【鎖 1】React UI メニュー（開発・Vercel デモ用）
  ボタンクリック → Application 層（projectManager）に直結

【鎖 2】Electron ネイティブメニュー（製品・.dmg 用）
  Menu click → sendToRenderer() のみ
  → App.tsx が geoAPI 経由で Application 層を呼ぶ
  → main.js IPC ハンドラーが fs / dialog の実体を提供する

2つの鎖が同じ Application 層（projectManager）を参照する
```

## 1. 開発の基本方針

### 1.1 SDD × CDD 原則

- **SDD（Spec-Driven Development）**: 実装前に必ず `docs/spec/` の spec ファイルを読む。仕様変更はコードより先に spec を修正する
- **CDD（Compiler-Driven Development）**: 完了条件は `pnpm tsc --noEmit`（型エラーゼロ）+ `pnpm test --run`（全テストグリーン）の両方通過
- **`any` 禁止**: 型エラーは自律修正する
- **プラン先行**: 実装前にプランを提示・承認を得てから実装開始

### 1.2 開発サイクル

```
1. docs/spec/[機能].spec.md を確認（または新規作成）
2. プランを提示・承認を得てから実装開始
3. 実装 → pnpm tsc --noEmit → 型エラーがあれば自律修正
4. pnpm test --run → 全グリーンになるまでループ
5. docs/progress/[task].log.md に完了を記録
6. セッション終了時に git commit + Day タグ（git tag dayN && git push origin dayN）
7. HANDOVER.md を更新
8. Obsidian dev-log に記録
```

### 1.3 ツール役割分担

## 4. 確立済みアーキテクチャ原則

### 4.1 engine.ts の制約（MUST）

- `engine.ts` は `App.tsx` に依存してはいけない・単体で動作できること
- Parameter Store の変更は必ず Command 経由のみ（直接代入禁止）
- Plugin には `renderer`・`enabled` フィールドを持たせること

### 4.2 Plugin 二分類（Day37 確立）

```
ModulatablePlugin（MIDI 2.0 / MacroKnob 制御可能）
  継承: GeometryPlugin / FXPlugin / ParticlePlugin / LightPlugin / CameraPlugin / SequencerPlugin（予定）

PluginBase のみ（外部制御不要）
  継承: TransitionPlugin / WindowPlugin / MixerPlugin
```

### 4.3 Transport Architecture（Day58 確立）

```
【外側の世界】プロトコル・セマンティクス
  cc-mapping.md（人間・AI が読む）
  MidiInputWrapper（MIDI → TransportEvent に変換）
  将来: OscInputWrapper / AI Layer（自然言語 → TransportEvent）

【境界線】
  TransportEvent { slot, value, source?, time? }

【コア層】純粋な値の処理
  TransportManager（slot + value を受け取るだけ）
  ParameterStore（slot番号をキーとして保持）
  Engine（CC番号の意味を知らない）
  TransportRegistry（slot→paramId の対応表）

【UI 層】鏡
  App.tsx（TransportRegistry を購読するだけ）
  WindowPlugin（availableParameters を filter するだけ）
```

**全入力経路の統一:**
```typescript
// 全ての入力源が同じ入り口を通る
engine.handleMidiCC(TransportEvent)
```

**Step 4（未完）: engine の ccMapService 依存解消**
- `engine.ts` の `flushParameterStore()` が `ccMapService` を直接参照している
- これが解消されると「外側と内側の境界線」が完全に閉じる
- 影響範囲が大きいため単独セッションで設計してから実装

### 4.4 MIDI 設計原則（Day42〜58 確立）

- **外部コントローラー受信**: Web MIDI API → `MidiInputWrapper` → `engine.handleMidiCC(TransportEvent)`
- **内部バス**: `TransportEvent { slot, value: 0.0〜1.0 }` で統一（プロトコル非依存）
- **MidiInputWrapper**: `src/drivers/input/MidiInputWrapper.ts`・シングルトン `midiInputWrapper`

### 4.5 Camera Plugin 設計原則（Day44〜45 確立）

- Camera は **独立した Plugin**（Geometry に埋め込まない）
- レイヤーごとに独立したインスタンス（`getCameraPlugin()` がファクトリ呼び出し）
- `isCameraUserOverridden` フラグで自動連動とユーザー意図を両立

### 4.6 Preset 設計原則（Day47 確立）

- Preset = `GeoGraphyProject` をそのまま保存
- 保存先: `localStorage`（キー: `geography:presets-v1`）
- Load → 即 engine 反映（APPLY 不要）

### 4.7 requiresRebuild フラグ（Day46 確立）

- メッシュの頂点数・形状が変わる param には `requiresRebuild: true` を必ず設定
- `engine.setGeometryParam()` が検知して `layerManager.rebuildPlugin()` を呼ぶ（destroy → create）

### 4.8 Clock の2系統（MUST・混同禁止）

```typescript
engine.threeClock  // THREE.Clock → getDelta() でフレーム間経過時間
engine.clock       // Clock（独自）→ BPM・beat 値管理・readonly で外部公開
```

### 4.9 FX 実装ルール（MUST）

- `createFxPlugins()` ファクトリを使う（シングルトン共有禁止）
- `fxStack.update()` は `enabled=false` でもスキップしない（`pass.enabled` で制御）
- FX スタック順序は `FX_STACK_ORDER` 定数に従う

### 4.10 Window / Panel 命名原則（Day28 確立）

- **Window** = Plugin エコシステムの UI・コントリビューターがデザインできる
- **Panel** = アプリ固定の小窓（Preferences Panel など）・コントリビューターが触れない
- **Simple Window** = 各 Plugin に付属するデフォルト最小 UI・カスタム Window Plugin がないときのフォールバック

### 4.11 Electron 薄い鎖アーキテクチャ（Day77〜78 確立）

```
【鎖 1】React UI メニュー（開発・Vercel デモ用）

  → main.js IPC ハンドラーが fs / dialog の実体を提供する

2つの鎖が同じ Application 層（projectManager）を参照する
```

## 12. ハマりポイント集（Day1〜58 の知見）

| 問題 | 原因 | 解決策 |
|---|---|---|
| FX ON/OFF が効かない | `fxStack.update()` で enabled=false をスキップ | `update()` は常に全 Plugin を呼ぶ・`pass.enabled` で制御 |
| 複数レイヤーで FX クラッシュ | `getAllFxPlugins()` シングルトンを共有 | `createFxPlugins()` ファクトリで毎回新インスタンス生成 |
| `IFxStack` 循環参照 | `types/index.ts` から `core/fxStack.ts` を import | `IFxStack` インターフェースを `types/index.ts` 内に定義 |
| preload.js が ES Module エラー | ルートに `"type": "module"` があるため | `electron/package.json` に `{ "type": "commonjs" }` を追加 |
| isDev 判定が誤る | `process.env.NODE_ENV` は Electron ビルドで信頼できない | `!app.isPackaged` を使う |
| beat 値が急ジャンプ | `setTempo()` 時に `startTime` をリセットしていない | テンポ変更時に `startTime = performance.now()` でリセット |
| drawImage で WebGL canvas が真っ黒 | WebGL のデフォルトは描画後バッファをクリア | `THREE.WebGLRenderer` に `preserveDrawingBuffer: true` を追加 |
| edit_file の oldText がマッチしない | macOS APFS が日本語を NFD で保存するが Claude は NFC で送る | `python3 /Users/shinbigan/nfc_normalize.py` で NFC 正規化 |
| Camera 初期値が常に Static になる | PreferencesPanel の useEffect が resolveCamId を使っていた | engine.getLayers() から cameraPlugin.id を直接読む（Day48修正済み） |
| Camera Plugin が複数レイヤーで状態干渉 | ファクトリを呼ばずシングルトンを共有 | `getCameraPlugin()` がファクトリ呼び出しで毎回独立インスタンス |
| Geometry param 変更で見た目が変わらない | segments 等はメッシュ再構築が必要 | `requiresRebuild: true` フラグ + `rebuildPlugin()` で destroy→create |
| MidiInputWrapper を App.tsx に直書きしていた | 関心の分離ができていなかった | `src/drivers/input/MidiInputWrapper.ts` に切り出し（Day58 Step2） |
| 200ms ポーリングで Plugin 数増加時に負荷増大 | 変化がなくても毎回全走査 | `engine.onParamChanged` コールバック + 16ms throttle に変更（Day58 Step3） |
| zsh でインラインコメント（#）がエラー | zsh のインタラクティブモードはコメント非対応 | コメント行とコマンド行を分けて渡す |
