## 1. プロジェクト概要

### 1.1 コンセプト

GeoGraphy は「No-Texture・Plugin駆動・マルチライブラリ対応」の映像制作プラットフォームです。

単なる VJ ツールではなく、複数の描画ライブラリを Plugin として束ねるプラットフォームとして設計されています。Three.js・PixiJS・opentype.js など、異なるライブラリを Plugin Group として追加でき、ユーザーは必要な Plugin だけを有効化して自分専用の最小構成で起動できます。

アプリ名は「Geometry（幾何学）+ 地形（Geo）」と「Graph（グラフ・図形）+ 描写（Graphy）」のダブルミーニングです。

### 1.2 ビジュアルコンセプト

- レトロフューチャー・シンセウェーブ
- ワイヤーフレーム・グリッド・幾何学形状
- テクスチャを一切使わない・計算とベクターだけで完結
- 暗背景 + 発光するライン・形状・面

### 1.3 Goals

- Three.js によるリアルタイム 3D レンダリング
- Electron アプリとしてネイティブ配布（.dmg）
- Plugin Group 単位でライブラリを追加できるプラットフォーム設計
- CLI ツールで新規 Plugin を自動スキャフォールド
- MIDI コントローラーによるリアルタイムパラメーター操作
- Resolume Arena との連携を前提とした設計
- Program / Preview バス構造による本格的な映像ミキシング
- MixerPlugin による操作 UI の完全カスタマイズ（v2〜）
- Transition Plugin による表現変形エンジン
- Geometry Launcher・Sequencer による自動 VJ 運用（v2〜）
- Embed Export：映像を Web サイト・スクリーンセーバーに埋め込む（v2〜）
- OSS として公開・世界中のコントリビューターが Plugin を追加できる

### 1.4 非機能要件・対象外

**全バージョン対象外（変更不可原則）：**
- テクスチャ貼り付け（No-Texture 原則・全バージョン禁止）
- 動画再生（Resolume に任せる）
- クラウド同期・コラボレーション
- Tauri / ネイティブアプリ化（Electron で解決済み）
- スマートフォン最適化（v2 以降）

**将来検討（v4 以降）：**
- Claude.ai 自動トランジション
- Video Input Driver（Resolume 受信）

## 3. Electron アーキテクチャ（Day24〜27 確立）

### 3.1 採用理由

| 比較 | Web ホスティング | Electron |
|---|---|---|
| サーバー | 必要 | **不要** |
| 配布方法 | URL | **.dmg / .exe をダウンロード** |
| ファイル操作 | ブラウザ制限あり | **Node.js fs で自由** |
| プラグイン追加 | 困難 | **外部フォルダをスキャン可能** |
| メニューバー | なし | **ネイティブメニューバー** |

### 3.2 プロセス構成

```
electron/
  main.js       ← メインプロセス（ウィンドウ生成・ファイルアクセス・メニュー）
  preload.js    ← レンダラーと main の橋渡し（IPC・contextBridge）
src/            ← 既存 Vite + React コードそのまま
```

セキュリティ設定（必須）：
- `nodeIntegration: false`
- `contextIsolation: true`
- `sandbox: true`

### 3.3 Electron 固定部分（レンダラーが触れない）

| 処理 | 場所 | 理由 |
|---|---|---|
| メニューバーの定義（GeoGraphy / File / View） | main.js | Electron `Menu` API |
| ファイルの Save / Load / Save As... | main.js | Node.js fs 操作 |
| ネイティブダイアログ（open/save） | main.js | Electron `dialog` API |
| Recent ファイルの管理・永続化 | main.js | ファイルシステム管理 |
| 外部プロセス連携・ファイル書き出し | main.js | Node.js 領域 |

### 3.4 レンダラー（React）に残すべき処理

| 処理 | 理由 |
|---|---|
| engine への操作（restoreProject / buildProject） | Three.js と密結合 |
| UI 状態管理 | React の責務 |
| Web MIDI API | ブラウザ API |
| autosave の送受信ロジック（useAutosave.ts） | engine と密結合 |

### 3.5 メニューバー設計

| メニュー | 項目 |
|---|---|
| GeoGraphy | About / Preferences...（⌘,）/ Services / Hide / Quit |
| File | New（⌘N）/ Open...（⌘O）/ Open Recent > / Save（⌘S）/ Save As...（⌘⇧S）/ Close Window |
| **View** | **全 Window の表示/非表示トグル（後述 §5）** |

### 3.6 将来実装時の注意（Output タブ）

Output タブで外部出力（録画・NDI・Syphon など）を実装する際は、最初から main.js 経由（IPC）で設計すること。レンダラーに Node.js 操作を直書きしない。

### 3.7 isDev 判定（MUST）

```javascript
// ✅ 正しい判定
const isDev = !app.isPackaged

// ❌ 使用禁止（Electron ビルドで信頼できない）
// const isDev = process.env.NODE_ENV !== 'production'
```

### 3.8 electron/ ディレクトリは CommonJS

`electron/package.json` に `{ "type": "commonjs" }` を置く。ルートの `"type": "module"` と共存するための必須対応。

### 3.9 プラグインの外部化（段階的導入）

| v1 | `src/plugins/` 内の固定プラグインのみ |

## 5. Window / Panel 設計（v1.9 新設）

### 5.1 命名原則

| 名称 | 定義 | 例 |
|---|---|---|
| **Window** | Plugin エコシステムの UI・コントリビューターがデザインできる | Mixer Simple Window / FX Simple Window / Window Plugin |
| **Panel** | アプリ固定の小窓・設定や選択のための UI・コントリビューターが触れない | Preferences Panel |

### 5.2 Simple Window（v1.9 新概念）

各 Plugin にデフォルトで付属する最小 UI。フォールバックとして機能する。

- **カスタム Window Plugin が有効なとき** → Simple Window は非表示
- **カスタム Window Plugin がないとき** → Simple Window が表示される

| Simple Window 名 | 対応 Plugin | 旧名称 |
|---|---|---|
| Mixer Simple Window | MixerPlugin | SimpleMixer |
| FX Simple Window | FXPlugin | FxControlPanel |
| Macro Knob Simple Window | MacroKnobManager | MacroKnobPanel |

### 5.3 Window Plugin（v1.9 再定義）

各 Plugin に対してカスタム UI を差し込める仕組み。

- 事前にどの Plugin にデータを渡すかを宣言して設計する
- コントリビューターが開発・PR で追加できる
- 有効化されると対応する Simple Window は非表示になる

### 5.4 View メニュー

すべての Window は View メニューから表示/非表示を切り替えられる。キーボードショートカット（§12）とも連動する。

```
View
  ├── Mixer Simple Window   （⌘1）
  ├── FX Simple Window      （⌘2）
  ├── Macro Knob Simple Window（⌘3）
  ├── ─────────────────────
  ├── Hide All Windows      （H）
  └── Show All Windows      （S）
```

## 8. プラットフォームアーキテクチャ

### 8.1 基本思想

GeoGraphy は「複数の描画ライブラリを Plugin として束ねるプラットフォーム」。エンジンは各 Plugin Group のライブラリを知らず、共通 Interface を通じてのみ通信する。

### 8.2 Plugin Group の概念

| Plugin Group | ライブラリ | バージョン | 主な用途 |
|---|---|---|---|
| Three.js Group | Three.js | v1 | 3D幾何学・ワイヤーフレーム・VJ映像 |
| PixiJS Group | PixiJS | v2 | 2D高速描画・フラットシェーディング |
| Typography Group | opentype.js | v3 | フォントアウトライン・Extrude・文字アニメーション |

### 8.3 PluginBase Interface（全 Plugin 共通）

```typescript
interface PluginBase {
  id: string          // Plugin の一意識別子
  name: string        // 表示名
  renderer: 'threejs' | 'pixijs' | 'opentype' | string  // 使用ライブラリ
  enabled: boolean    // Plugin の有効・無効
}
```

### 8.4 Plugin 自動登録

```typescript
const modules = import.meta.glob('./*/index.ts', { eager: true });
Object.values(modules).forEach((m: any) => registry.register(m.default));
```

## 9. Plugin / Driver 設計

### 9.1 全種類

| 種類 | 役割 | UI | v1 |
|---|---|---|---|
| Geometry Plugin | 何を描画するか（3D幾何学） | Simple Window / Window Plugin | ✅ |
| Particle Plugin | 背景・雰囲気（パーティクル） | Simple Window / Window Plugin | ✅ |
| FX Plugin | エフェクト（Bloom・グリッチ等） | FX Simple Window / Window Plugin | ✅ |
| Light Plugin | ライティング（環境光・ポイントライト等） | Simple Window / Window Plugin | ✅ |
| 2D Shape Plugin | 2D 塗り潰し図形（PixiJS） | Simple Window / Window Plugin | v2 |
| Typography Plugin | フォント・Extrude（opentype.js） | Simple Window / Window Plugin | v3 |
| **MixerPlugin** | Program/Preview のミキシング操作 | Mixer Simple Window / Window Plugin | v1固定→v2Plugin化 |
| **Transition Plugin** | 映像・パラメーターの変形（UI なし・処理のみ） | UI なし | v1最小実装 |
| **Window Plugin** | 各 Plugin へのカスタム UI・事前にデータ渡し先を宣言 | それ自体が UI | v2〜 |
| Tempo Driver | テンポ入力（Ableton Link・タップ・手動） | なし | ✅ |
| Input Driver | デバイス入力（MIDI・トラックボール・ゲームパッド） | なし | ✅ |
| Output Driver | 出力先（HDMI・Resolume・NDI） | なし | ✅ |
| Modulator Driver | パラメーター値の入力源（LFO・エンベロープ等） | なし | ✅ |
| Video Input Driver | 外部映像をレイヤーとして取り込む | なし | v4 |

> ※ v1 は Mixer Simple Window（固定実装）+ Beat Cut / CrossFade（最小 Transition）のみ。MixerPlugin の交換・追加は v2 から。Window Plugin の追加は v2 から。

### 9.2 エンジン固定部分（Plugin が触れない）

| 固定部分 | 役割 |
|---|---|
| Parameter Store | 全パラメーターの一元管理 |
| Plugin Registry | Plugin の登録・切り替え・list() |
| Command パターン | アンドゥ・リドゥ |
| レンダリングループ | requestAnimationFrame・delta・beat の供給 |
| BPM クロック | Tempo Driver からの BPM・beat 値の管理 |
| MacroKnobManager | MIDI CC とパラメーターの対応管理 |

### 9.3 Single Source of Truth

すべてのパラメーターは Parameter Store で一元管理。MIDI・GUI・AUTO のどこから変更しても同じ値が更新される。

## 10. Clock 設計（Day10 確立・MUST）

engine には2種類の Clock が存在する。**混同禁止。**

| フィールド | クラス | 用途 |
|---|---|---|
| `engine.threeClock` | `THREE.Clock` | `getDelta()` でフレーム間の経過時間を取得 |
| `engine.clock` | `Clock`（独自） | BPM・beat 値の管理・外部から `readonly` で公開 |

```typescript
// BPM Clock の beat 値（0.0〜1.0 で繰り返す）
const beat = (elapsed / beatDuration) % 1

// テンポ変更時は startTime をリセット（急ジャンプ防止）
setTempo(bpm): void {
  this.startTime = performance.now()
  this.bpm = bpm
}
```

## 11. Program / Preview バス構造（Day25 実装済み）

### 11.1 概要

| バス | 役割 | 実体 |
|---|---|---|
| Program バス | 今出力中の映像・Resolume 等に送られる | フルサイズ Three.js Scene |
| Preview バス | 次に出す映像・エディット中 | SceneState（JSON）+ 小サムネイルキャンバス（320×180） |

### 11.2 Preview バスのメモリー設計（MUST）

- MUST: Preview バスは `THREE.Scene` / `THREE.WebGLRenderer` を持たない
- Three.js オブジェクトは Program に昇格した瞬間に生成・切り替え後に旧 Program を `dispose()`
- サムネイル確認用に小さなキャンバス（320×180）を1つだけ持つ
- SceneState の変更は Command パターン経由

### 11.3 SceneState Interface

```typescript
interface SceneState {
  layers: LayerState[]
}

interface LayerState {
  geometryId: string
  geometryParams: Record<string, number>
  fxStack: FxState[]
  opacity: number
  blendMode: string
}
```

### 11.4 切り替えフロー

```
Preview バスで次の映像を準備・エディット
  ↓
Mixer Window でトランジションを選択（プルダウン）
  ↓
クロスフェーダーで切り替え実行（MIDI フェーダーまたはマウス）
  ↓
Transition Plugin が SceneState を変形しながら移行
  ↓
Preview が Program に昇格・旧 Program が Preview に降格
```

### 11.5 モニタリングモード

| モード | 表示 |
|---|---|
| 通常モード | Program 大・Preview 小（サムネイル） |
| 準備モード | Program 小・Preview 大 |
| デュアルモニター | モニター1：Program フル / モニター2：Preview フル |

### 17.1 v1 実装 FX（10種）

| FX | 種類 | デフォルト |
|---|---|---|
| AfterImage | 合成系 | ON（damp 0.85） |
| Feedback | 合成系 | OFF（amount 0.7） |

| CRT | レトロ系 | OFF |
| Glitch | グリッチ系 | OFF |
| ColorGrading | 色調整 | ON（saturation 1.0 / contrast 1.0 / brightness 1.0） |

### 17.2 FX スタック順序（厳守）

```
AfterImage → Feedback → Bloom → Kaleidoscope → Mirror
→ ZoomBlur → RGBShift → CRT → Glitch → ColorGrading（必ず最後）
```

### 17.3 FX 実装の重要知見（Day16〜18 確立）

**シングルトン共有禁止（MUST）**
`createFxPlugins()` ファクトリ関数を使うこと。`getAllFxPlugins()` でシングルトンを共有すると複数レイヤーの `addPass` が同一 Pass に走りクラッシュする。

**update() のスキップ禁止（MUST）**
`fxStack.update()` は `enabled=false` の Plugin もスキップしないこと。`pass.enabled` で制御する。

**レイヤー配置ルール**
- layer-1（背景）: particle（starfield）・FX なし
- layer-2（主役）: geometry + FX スタック

## 22. マクロノブ（Day13〜14 実装済み）

### 22.1 MacroKnobManager（エンジンコア）

MIDI CC とパラメーターの対応管理を担うコア。Plugin が触れないエンジン固定部分。

```
MIDI機器（物理ノブ）
  ↓ CC番号
Input Driver（MIDI Driver）
  ↓
MacroKnobManager（src/core/macroKnob.ts）← エンジン固定部分
  ↓ normalize() して
Parameter Store → Command 経由でパラメーター更新
```

### 22.2 Macro Knob Simple Window（UI）

MacroKnobManager の設定・確認用 UI。View メニューから表示/非表示を切り替えられる。

| 項目 | 仕様 |
|---|---|
| ノブ数 | 32個（8ノブ × 4行）・アコーディオンで折りたたみ |
| 1ノブの最大割り当て | 3パラメーターまで（超えると `setKnob()` がエラー throw） |
| MIDI 範囲 | 0〜127 |
| 正規化 | `min + (midi / 127) * (max - min)`（`normalize()` ピュア関数） |
| カーブ | v1：linear / v2：exp・log・s-curve |
| 設定方法 | ノブ右クリック → EditDialog（名前・MIDI CC 入力） |
