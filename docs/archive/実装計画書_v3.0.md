## 1. 開発の基本方針

### 1.1 SDD × CDD 原則

- **SDD（Spec-Driven Development）**: 実装前に必ず `docs/spec/` の spec ファイルを読む。仕様変更はコードより先に spec を修正する
- **CDD（Compiler-Driven Development）**: 完了条件は `pnpm tsc --noEmit`（型エラーゼロ）+ `pnpm test --run`（全テストグリーン）の両方通過
- **`any` 禁止**: 型エラーは自律修正する
- **プラン先行**: 実装前にプランを提示・承認を得てから実装開始

### 1.2 開発サイクル

```
1. docs/spec/[機能].spec.md を確認（または新規作成）
2. プランを提示・承認を得てから実装開始
3. 実装 → pnpm tsc --noEmit → 型エラーがあれば自律修正
4. pnpm test --run → 全グリーンになるまでループ
5. docs/progress/[task].log.md に完了を記録
6. セッション終了時に git commit + Day タグ（git tag dayN && git push origin dayN）
7. HANDOVER.md を更新
8. Obsidian dev-log に記録
```

## 4. 確立済みアーキテクチャ原則

### 4.1 engine.ts の制約（MUST）

- `engine.ts` は `App.tsx` に依存してはいけない・単体で動作できること
- Parameter Store の変更は必ず Command 経由のみ（直接代入禁止）
- Plugin には `renderer`・`enabled` フィールドを持たせること

### 4.2 Clock の2系統（MUST・混同禁止）

```typescript
engine.threeClock  // THREE.Clock → getDelta() でフレーム間経過時間
engine.clock       // Clock（独自）→ BPM・beat 値管理・readonly で外部公開
```

### 4.3 FX 実装ルール（MUST）

- `createFxPlugins()` ファクトリを使う（シングルトン共有禁止）
- `fxStack.update()` は `enabled=false` でもスキップしない（`pass.enabled` で制御）
- FX スタック順序は `FX_STACK_ORDER` 定数に従う

### 4.4 Electron アーキテクチャ（MUST）

- Save / Load / Recent / メニュー定義 → `main.js` に集約
- engine 操作 / Web MIDI / autosave → レンダラー（React）
- Output タブ実装時は最初から IPC 経由で設計する
- `isDev = !app.isPackaged`（`process.env.NODE_ENV` は使わない）

### 4.5 SimpleMixer・MacroKnobPanel（MUST）

- 閉じるボタンを実装してはいけない（常時表示）
- v1 の時点から MixerPlugin Interface に準拠した実装にすること

### 4.6 Transition Plugin（MUST）

- `execute()` は純粋関数として実装（副作用なし・戻り値は `SceneState`）
- CrossFade の category は `'parameter'`

## 5. ハマりポイント集（Day1〜27 の知見）

| 問題 | 原因 | 解決策 |
|---|---|---|
| FX ON/OFF が効かない | `fxStack.update()` で enabled=false をスキップ → `pass.enabled` が true のまま | `update()` は常に全 Plugin を呼ぶ・`pass.enabled` で制御 |
| 複数レイヤーで FX クラッシュ | `getAllFxPlugins()` シングルトンを共有 → 同一 Pass に複数 addPass | `createFxPlugins()` ファクトリで毎回新インスタンス生成 |
| `IFxStack` 循環参照 | `types/index.ts` から `core/fxStack.ts` を import | `IFxStack` インターフェースを `types/index.ts` 内に定義 |
| App.tsx が engine を呼んでいなかった | 旧 App.tsx が独自 Scene/Renderer を管理していた | `engine.initialize(container)` → `engine.start()` のみに変更 |
| preload.js が ES Module エラー | ルートに `"type": "module"` があるため | `electron/package.json` に `{ "type": "commonjs" }` を追加 |
| isDev 判定が誤る | `process.env.NODE_ENV` は Electron ビルドで信頼できない | `!app.isPackaged` を使う |
| `dist-electron/` が誤コミット | `.gitignore` に未追加 | `.gitignore` に追加・`git rm -r --cached` で追跡解除 |
| zsh が `!app.isPackaged` をエラー扱い | zsh の history expansion | filesystem ツールで直接ファイル編集 |
| beat 値が急ジャンプ | `setTempo()` 時に `startTime` をリセットしていない | テンポ変更時に `startTime = performance.now()` でリセット |
| THREE.Clock と BPM Clock の命名衝突 | 同じ名前で管理していた | `threeClock`（delta 用）と `clock`（BPM 用）に明確分離 |
| GeoGraphyProject の layers フィールド不整合 | `sceneState.layers` / `setup` / `version` の構造を誤解 | 正しい構造で初期化 |

### 10.2 Geometry ランチャーの実装方針

```typescript
const playlist: SceneState[] = [sceneA, sceneB, sceneC]
let current = 0

onBeat(4, () => {
  previewBus.setState(playlist[current])
  mixer.triggerCrossfade(selectedTransition)
  current = (current + 1) % playlist.length
})
```

SceneState のリストを順番に Preview バスに送り、クロスフェーダーを自動実行するだけ。

## 11. 注意点・矛盾点の整理

| 懸念点 | 解決方針 |
