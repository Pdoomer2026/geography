## 1. プロジェクト概要

### 1.1 コンセプト

GeoGraphy は「No-Texture・Plugin駆動・マルチライブラリ対応」の映像制作プラットフォームです。

単なる VJ ツールではなく、複数の描画ライブラリを Plugin として束ねるプラットフォームとして設計されています。Three.js・opentype.js など、異なるライブラリを Plugin Group として追加でき、ユーザーは必要な Plugin だけを有効化して自分専用の最小構成で起動できます。

アプリ名は「Geometry（幾何学）+ 地形（Geo）」と「Graph（グラフ・図形）+ 描写（Graphy）」のダブルミーニングです。

### 1.2 ビジュアルコンセプト

- レトロフューチャー・シンセウェーブ
- ワイヤーフレーム・グリッド・幾何学形状
- テクスチャを一切使わない・計算とベクターだけで完結
- 暗背景 + 発光するライン・形状・面

### 1.3 Goals

- Three.js によるリアルタイム 3D レンダリング
- Electron アプリとしてネイティブ配布（.dmg）
- Plugin Group 単位でライブラリを追加できるプラットフォーム設計
- CLI ツールで新規 Plugin を自動スキャフォールド
- MIDI コントローラーによるリアルタイムパラメーター操作
- DAW の IAC（Inter-Application Communication）によるリアルタイムパラメーター操作（v1〜）
- Resolume Arena との連携を前提とした設計
- Program / Preview バス構造による本格的な映像ミキシング
- MixerPlugin による操作 UI の完全カスタマイズ（v1〜）
- Transition Plugin による表現変形エンジン
- Geometry Launcher・Sequencer による自動 VJ 運用（v1〜）
- Embed Export：映像を Web サイト・スクリーンセーバーに埋め込む（v2〜）
- OSS として公開・世界中のコントリビューターが Plugin を追加できる（v2〜）

### 1.4 非機能要件・対象外

**全バージョン対象外（変更不可原則）：**
- テクスチャ貼り付け（No-Texture 原則・全バージョン禁止）
- 動画再生（Resolume に任せる）
- クラウド同期・コラボレーション
- Tauri / ネイティブアプリ化（Electron で解決済み）
- スマートフォン最適化（v2 以降）

**将来検討（v4 以降）：**
- Claude.ai 自動トランジション
- Video Input Driver（Resolume 受信）

## 3. Electron アーキテクチャ（Day24〜27 確立）

### 3.1 採用理由

| 比較 | Web ホスティング | Electron |
|---|---|---|
| サーバー | 必要 | **不要** |
| 配布方法 | URL | **.dmg / .exe をダウンロード** |
| ファイル操作 | ブラウザ制限あり | **Node.js fs で自由** |
| プラグイン追加 | 困難 | **外部フォルダをスキャン可能** |
| メニューバー | なし | **ネイティブメニューバー** |

### 3.2 プロセス構成

```
electron/
  main.js       <- メインプロセス（ウィンドウ生成・ファイルアクセス・メニュー）
  preload.js    <- レンダラーと main の橋渡し（IPC・contextBridge）
src/            <- Vite + React コード
```

セキュリティ設定（必須）：
- `nodeIntegration: false`
- `contextIsolation: true`
- `sandbox: true`

### 3.3 Electron 固定部分（レンダラーが触れない）

> Day78 確定・⚠️ 要確認マーク解除

**「実装の置き場所」は main.js。ただしメニューハンドラーは薄い鏡（→ §3.3.1）。**

| 処理 | 場所 | 理由 |
|---|---|---|
| メニューバーの定義（GeoGraphy / File / View） | main.js | Electron `Menu` API |
| ネイティブダイアログの実体（open / save） | main.js（IPC ハンドラー） | Electron `dialog` API |
| ファイルの fs 操作（readFile / writeFile） | main.js（IPC ハンドラー） | Node.js fs 操作 |
| Recent の永続化・buildMenu() | main.js（IPC ハンドラー） | ファイルシステム管理 |
| autosave / save-recording / get-data-dir | main.js（IPC ハンドラー） | Node.js fs 操作 |
| cc-map / cc-overrides 読み書き | main.js（IPC ハンドラー） | Node.js fs 操作 |
| 外部プロセス連携・ファイル書き出し | main.js | Node.js 領域 |

#### 3.3.1 メニューハンドラーは「薄い鏡」（Day78 確立）

メニューハンドラーは `sendToRenderer(channel)` を呼ぶだけ。dialog / fs 操作は行わない。

```
Menu click → sendToRenderer('menu:open') のみ
          → renderer が dialog → loadFile → addRecent を順に実行
          → main.js IPC ハンドラーが fs / dialog の実体を提供する
```

#### 3.3.2 2つの薄い鏡（Day77 設計確定）

```
【鏡 1】React UI メニュー（開発・Vercel デモ用）
  ボタンクリック → Application 層（projectManager）に直結

【鏡 2】Electron ネイティブメニュー（製品・.dmg 用）
  Menu click → sendToRenderer() のみ
  → App.tsx が geoAPI 経由で Application 層を呼ぶ

2 つの鏡が同じ Application 層（projectManager）を参照する
```

### 3.4 レンダラー（React）に残すべき処理

> Day78 確定・⚠️ 要確認マーク解除

**「制御層・オーケストレーション」はレンダラーが持つ。**

| 処理 | 理由 |
|---|---|
| engine への操作（restoreProject / buildProject） | Three.js と密結合 |
| UI 状態管理（WindowMode 等） | React の責務 |

| キーボードショートカット管理 | UI イベント |
| 録画ロジック（startRecording / stopRecording） | engine と密結合 |
| MidiMonitor コールバック管理 | UI コンポーネントとの接続 |
| Open / SaveAs / OpenRecent のオーケストレーション | dialog → fs → projectManager の順序制御（Day78〜） |
| addRecent の呼び出しタイミング管理 | ファイルを開いた後の副作用管理（Day78〜） |

## 8. プラットフォームアーキテクチャ（Day67〜68 確定・3ゾーン構造）

### 8.1 3ゾーン構造

Day67〜68 の大規模リファクタリングにより、コードベースを3つのゾーンに分離した。

```
src/
  engine/       Three.js プラグイン（疎結合・描画のみ）
  application/  意思決定・Registry・Transport・型定義
  ui/           表示・操作（React コンポーネント）
```

| ゾーン | 責務 | import ルール |
|---|---|---|
| engine/ | 描画・Plugin 定義 | application/ の型のみ依存 |
| application/ | 全ロジック・Registry | ui/ に依存しない |
| ui/ | React コンポーネント | engine/ + application/ を参照可 |

### 8.2 Plugin Group の概念

| Plugin Group | ライブラリ | バージョン | 主な用途 |
|---|---|---|---|
| Three.js Group | Three.js | v1 | 3D幾何学・ワイヤーフレーム・VJ映像 |
| Typography Group | opentype.js | v1 | フォントアウトライン・Extrude・文字アニメーション |

### 8.3 PluginBase Interface（全 Plugin 共通）

```typescript
interface PluginBase {
  id: string
  name: string
  renderer: 'threejs' | 'opentype' | string
  enabled: boolean
}
```

### 8.4 Plugin 二分類（Day37 確立）

```
ModulatablePlugin（MIDI 2.0 / MacroKnob 制御可能）
  継承: GeometryPlugin / FXPlugin / ParticlePlugin / LightPlugin / CameraPlugin

PluginBase のみ（外部制御不要）
  継承: TransitionPlugin / WindowPlugin / MixerPlugin
```

### 8.5 Plugin 自動登録

```typescript
const modules = import.meta.glob('./*/index.ts', { eager: true })
Object.values(modules).forEach((m) => registry.register(m.default))
```

## 10. Transport Architecture（Day58〜76 確立）

### 10.1 設計原則

GeoGraphy の全パラメーター受け渡しは **TransportEvent** フォーマットに統一されている。プロトコルの差異は Input Wrapper が吸収し、Engine はプロトコルを一切知らない。

```
【外側の世界】プロトコル・セマンティクス
  cc-mapping.md（人間・AI が読む）
  MidiInputWrapper（MIDI → TransportEvent に変換）
  将来: OscInputWrapper / AI Layer

【境界線】
  TransportEvent { slot, value, source?, time? }

【コア層】純粋な値の処理（自律的に動く）
  TransportManager（slot + value を受け取るだけ）
  ParameterStore（GeoParamAddress をキーとして保持）
  Engine（CC番号の意味を知らない）
  TransportRegistry（slot→paramId の対応表）

【UI 層】鏡
  App.tsx（TransportRegistry を購読するだけ）
  Window（availableParameters を filter するだけ）
```

### 10.2 slot エンコーディング（Day76 確定・SSoT）

```
CC 空間（slot 0〜2047）:
  slot = channel * 128 + cc

  ch0, CC7  -> slot 7     Track Fader 1（APC40）
  ch1, CC7  -> slot 135   Track Fader 2
  ch0, CC48 -> slot 48    Track Knob 1

Note 空間（slot 2048〜4095）:
  slot = 2048 + channel * 128 + note

  ch0, Note 82 -> slot 2130   Scene Launch 1
  ch0, Note 52 -> slot 2100   Clip Stop Track 1
```

MIDI 1.0 の全アドレス空間（16ch × 128CC）と Note 空間を完全分離・衝突ゼロ。
